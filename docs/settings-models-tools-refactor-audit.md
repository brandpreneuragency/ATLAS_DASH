# Settings Models/Tools Refactor Audit

Generated by Hermes orchestrator before the multi-agent refactor.

## 1. Current Settings tab/navigation structure

- Settings is a document-mode view, not a standalone route.
- `src/stores/uiStore.ts` defines `SettingsSubTab` as `models | actions | appearance | agents` and tracks `activeSettingsSubTab`.
- `src/components/header/moduleNav.ts` defines `SETTINGS_HEADER_TABS` with Models, Actions, Appearance, Agents.
- `src/components/header/TabBar.tsx` renders `SETTINGS_HEADER_TABS` when `activeView === 'settings'` and calls `setActiveSettingsSubTab(item.key)`.
- `src/components/settings/SettingsDocument.tsx` conditionally renders one Settings section per active sub-tab.
- Each section is expected to pass right-column slots into `SettingsAISidebar` so every sub-tab has a separate persisted Settings AI chat context.

## 2. Current ModelsSection responsibilities

`src/components/settings/ModelsSection.tsx` currently owns the Settings → Models three-column composition:

- Reads `providerConfigs` and `searchConfig` from `aiStore`.
- Builds left-panel groups for LLM, Web Search, Embeddings, and Vector Store.
- Mixes AI/model providers and search providers in the same left rail.
- Renders Exa and Tavily virtual search providers.
- Renders empty Embeddings/Vector groups as clickable placeholder buttons via `renderEmptyGroup`.
- Tracks `focusProviderId` locally and passes it to `ModelManagementContent`.
- Renders `ModelManagementContent isInline` in the center panel.
- Provides a left-panel `Connect provider` button that also leaves the center panel with another connect provider button.

## 3. Current ModelsContent responsibilities

`src/components/settings/ModelsContent.tsx` currently contains most model/provider settings logic:

- Loads provider API keys from `secureStorage`.
- Tracks draft provider configs, draft API keys, draft base URLs, hidden model IDs, import state, connection state, expanded accordion IDs, search provider drafts, and drawer state.
- Persists provider edits to Dexie via `saveProviderConfig` and credentials to `secureStorage`.
- Persists Exa/Tavily search keys through `saveSearchConfig`.
- Renders provider accordions using `ProviderAccordionItem`.
- Renders the add/connect provider drawer using `ConnectProviderDrawer`.
- Renders inline Exa/Tavily search-key panels when virtual search provider IDs are focused.
- Renders placeholder coming-soon sections for embeddings/vector groups.

## 4. Where Exa/Tavily/Firecrawl/Brave currently appear

- `SearchConfig` in `src/types/index.ts` supports `exaKey`, `tavilyKey`, `firecrawlKey`, `braveKey`, `enabled`, and `searchProvider`.
- `aiStore.defaultSearchConfig()` initializes all four keys and `searchProvider: 'tavily'`.
- `aiStore.loadAISettings()` loads all four from `db.settings`.
- `aiStore.saveSearchConfig()` persists all four keys plus `enabled` and `searchProvider`.
- Settings → Models UI currently only exposes Exa and Tavily in `ModelsSection.tsx` / `ModelsContent.tsx`.
- Firecrawl and Brave exist in state/types but are not surfaced in the current Models UI.

## 5. Provider status derivation points

- `src/types/index.ts` currently defines `ProviderStatus = 'connected' | 'not_connected' | 'needs_setup'`.
- `src/stores/aiStore.ts` has weak status helpers:
  - `deriveStatus(hasKey, _isCustom, hasBaseUrl)` always returns `not_connected` unless base URL/key are missing, where it also returns `not_connected`.
  - `refreshStatus(hasKey, hasBaseUrl, currentStatus)` preserves `connected` only if current status is already connected.
- `connectProvider()` marks a provider `connected` only after base URL, key, and at least one enabled model exist.
- `importProviderModels()` currently keeps status unchanged on success and sets `not_connected` on `ProviderImportError`.

## 6. Existing searchConfig/saveSearchConfig shape

From `src/types/index.ts`:

```ts
export type SearchProvider = 'tavily' | 'firecrawl' | 'brave' | 'exa';

export interface SearchConfig {
  exaKey: string;
  tavilyKey: string;
  firecrawlKey: string;
  braveKey: string;
  enabled: boolean;
  searchProvider: SearchProvider;
}
```

`saveSearchConfig(config)` in `src/stores/aiStore.ts` stores each field into `db.settings` keys:

- `exaKey`
- `tavilyKey`
- `firecrawlKey`
- `braveKey`
- `enabled`
- `searchProvider`

## 8. Existing commands for typecheck/lint/build/test

From `package.json`:

- `npm run build` = `tsc -b && vite build`
- `npm run lint` = `eslint .`
- No `typecheck` script exists.
- No `test` script exists.

Baseline results before refactor:

- `npm ci` failed because `package-lock.json` was out of sync with `package.json`.
- `npm install` succeeded and updated/installable dependencies, with peer dependency warnings for React 19 versus some Plasmic transitive packages and `npm audit` reporting 4 vulnerabilities.
- `npm run build` passed.
- `npm run lint` failed with 49 errors and 8 warnings before the refactor. Failures are broad and pre-existing, including React compiler rules, no-explicit-any, no-unused-vars, prefer-const, and fast-refresh export warnings.

## 9. Risks before parallel work starts

- Multiple planned agents may touch `ModelsContent.tsx`, `ModelsSection.tsx`, `aiStore.ts`, `types/index.ts`, `settings.css`, and i18n files; conflicts are likely if run fully in parallel.
- Existing `ModelsContent.tsx` combines provider connection, model import, search provider config, placeholder rendering, autosave, modal, and inline behavior; splitting it needs careful preservation of persistence semantics.
- Baseline lint is already red, so final quality gate should use `npm run build` as the hard compile gate and treat lint as pre-existing unless new files add obvious lint failures.
- `npm install` modified `package-lock.json` because it was out of sync; this should be reviewed separately from source refactor changes.
- Search provider secrets must stay in the existing storage pattern and must not be printed or copied into reports.
- Destructive provider deletion must require confirmation.
- Usage tab must not invent quota/cost/token data.
